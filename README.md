# TP-Integrador-Ecommerce-Obj-2
<div align="justify">
<h3>Presentación</h3>
El presente trabajo documenta las decisiones de diseño tomadas al modelar el sistema de gestión de pedidos, catálogo y pagos de una plataforma de e-commerce. A lo largo del desarrollo nos encontramos con distintos problemas de diseño recurrentes (comportamientos que varían según un estado, algoritmos intercambiables, estructuras jerárquicas de objetos, notificaciones desacopladas entre módulos) para los cuales aplicamos los patrones de diseño que mejor se ajustaban a cada contexto, priorizando siempre el bajo acoplamiento y el cumplimiento de los principios SOLID.
  <br>
En primer lugar, para el catálogo de productos se utilizó el patrón <strong>Composite</strong>, ya que el sistema debe representar productos individuales y paquetes que pueden contener otros paquetes. Donde el sistema pueda tratarlos con la misma interfaz.
    <br>
En segundo lugar, para el ciclo de vida del pedido utilizamos el patrón <strong>State</strong>, dado que el comportamiento del Pedido irá siendo modificado, dependiendo del estado en el que se encuentre.  <br>
En tercer lugar, utilizamos un <strong>Template Method</strong> para la validación de los métodos de pago, ya que estos requieren realizar una validación, donde cada método de pago requiere distintos datos y métodos pero donde todos tienen el mismo flujo general de procesamiento: validar, reservar fondos, ejecutar la transacción y notificar. Con el Template Method, definimos la estructura del algoritmo y cada paso se especifica en las subclases.  <br>
En cuarto lugar, para los metodos de envío se utilizó el patrón <strong>Strategy</strong>, ya que el método de envío es una elección del cliente. Estos encapsulan cada algoritmo de cálculo de costo y tiempo de entrega de las estrategias. Estas son intercambiables en tiempo de ejecución.  <br>
En quinto lugar, para el sistema de notificaciones utilizamos el patrón <strong>Observer</strong>, con el fin de desacoplar al Pedido de los distinto módulos que deben reaccionar a los cambios del Pedido.  <br>
En sexto lugar, para la búsqueda y filtro en el catálogo, se utilizó nuevamente el patrón <strong>Composite</strong>, dado que los criterios de búsqueda pueden ser simples o compuestos. Estos deben ser combinables de forma flexible y recursiva, sin que el cliente deba distinguir en sus clases.  <br>
En último lugar, para la generación de reportes se aplicó el patrón <strong>Visitor</strong>, ya que se necesitaba definir una nueva operación (el cálculo de productos más vendidos) sobre la estructura existente de Producto y Paquete sin modificar dichas clases ni acoplarlas a la lógica de reportería. De esta forma, cada tipo de reporte puede implementarse como un nuevo visitor independiente, mientras que la recolección de datos queda completamente separada de su posterior exportación a los distintos formatos de salida (texto plano, CSV, HTML).
